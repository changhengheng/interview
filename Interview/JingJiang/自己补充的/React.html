1. React 和 Vue 的区别 ​

1. 数据流 ​

  • Vue 默认支持双向数据绑定
  • React 提倡单向数据流

2. 虚拟 DOM ​

  • Vue 宣称可以更快地计算出 Virtual DOM 的差异，这是由于它在渲染过程中，会跟踪每一个组件的依赖关系，不需要重新渲染整个组件树。
  • 对于 React 而言，每当应用的状态被改变时，全部子组件都会重新渲染。当然，这可以通过 PureComponent/shouldComponentUpdate 这个生命周期方法来进行控制，但 Vue 将此视为默认的优化。

3. 组件化 ​

  • Vue 提供单文件组件方式，类似于 HTML 语法。
  • React 通过 JSX 语法来定义组件。

4. 更新用户界面方式 ​

  • Vue 的数据是响应式的，更新数据同时用户界面也会更新
  • React 要触发用户界面更新需要使用 this.setState 方法或者 useState 提供的函数

5. 构建工具 ​

  • Vue 有 vite-cli / vue-cli
  • React 有 create-react-app

6. 跨平台 ​

  • Vue 有 Weex
  • React 有 React Native

7. 数据可变性 ​

  • Vue 使用的是可变数据
  • React 强调不可变数据

2. 谈谈 redux ​

1. 概念 ​

集中式管理状态的方案

2. 组成 ​

  • store 所有的状态都储存在 store 中
  • actions 用来返回一个 action 对象函数模块
  • reducers 根据之前状态和 action 对象来计算生成新状态函数模块

3. 使用 ​

  • 读取：页面可以直接从 store 中读取数据使用
  • 更新：页面不能直接更新 store 中的数据，调用 actions 函数生成 action 对象，触发 reducer 函数计算生成新状态，更新 store 中的状态。

redux 和 mobx 的区别

  1. redux 将数据保存在单一的 store 中，mobx 将数据保存在分散的多个 store 中
  2. redux 使用 plain object 保存数据，更新数据后还需要手动处理用户界面的更新; mobx 使用 observable 保存数据，数据是响应式的，数据变化后自动处理用户界面的更新
  3. redux 使用不可变状态，这意味着状态是只读的，不能直接去修改它，而是应该返回一个新的状态，同时使用纯函数; mobx 中的状态是可变的，可以直接对其进行修改

3. 对 React Hook 的理解 ​

在 React 中定义组件有两种方式：类组件和函数式组件。它们区别：

  • 类组件需要继承 class，函数组件不需要；
  • 类组件可以访问生命周期方法，函数组件不能；
  • 类组件中可以获取到实例化后的 this，并基于这个 this 做各种各样的事情，而函数组件不可以；
  • 类组件中可以定义并维护 state（状态），而函数组件不可以；

React Hooks 是一套能够使函数组件更强大、更灵活的“钩子”。

  • useState 让函数组件定义并维护 state（状态）
  • useEffect 让函数组件使用生命周期（componentDidMount）
  • useCallback 缓存函数
  • useMemo 缓存计算结果值

4. React 组件间通信 ​

  1. props （父 -> 子）
  2. 回调函数（子 -> 父）
  3. 自定义事件通信（发布订阅模式）（兄弟）
  4. context（祖孙）
  5. redux / mobx（任意）

2. 函数式组件生命周期 ​

通常使用 React.useEffect 来完成

扩展：useEffect 依赖为空数组与 componentDidMount 区别在 render 执行之后，componentDidMount 会执行，如果在这个生命周期中再一次 setState ，会导致再次 render ，但是浏览器只会渲染第二次 render 返回的值，这样可以避免闪屏。 但是 useEffect 是在真实的 DOM 渲染之后才会去执行，这会造成两次 render ，这两次可能都会渲染出来，可能会闪屏。 实际上 useLayoutEffect 会更接近 componentDidMount 的表现，它们都同步执行且会阻碍真实的 DOM 渲染的。 扩展：useEffect 和 useLayoutEffect 区别对于 React 的函数组件来说，其更新过程大致分为以下步骤：

  1. 因为某个事件回调导致 state 发生变化。
  2. React 内部更新 state 变量。
  3. React 处理更新组件 render 方法中 return 出来的 DOM 节点（进行一系列 dom diff 、调度等流程）。
  4. 将更新过后的 DOM 数据绘制到浏览器中。
  5. 用户看到新的页面。

useEffect 在第 4 步之后执行，且是异步的，保证了不会阻塞浏览器进程。 useLayoutEffect 在第 3 步至第 4 步之间执行，且是同步代码，所以会阻塞后面代码的执行。

6. 谈谈 React Router ​

本文以 router-router-dom@6.8.2 为例

1. 概念 ​

用来开发单页面应用

2. 基本使用 ​

  • 通过createBrowserRouter创建路由对象，里面放路由配置项
    • path 路由路径
    • element 路由组件
  • 通过RouterProvider组件来渲染路由
tsx
import { createBrowserRouter } from "react-router-dom";

import Home from "../pages/home";
import About from "../pages/about";
import ErrorPage from "../pages/error-page";

const router = createBrowserRouter([
  {
    path: "/",
    element: <Home />,
    errorElement: <ErrorPage />, // 匹配404
  },
  {
    path: "/about",
    element: <About />,
  },
]);

export default router;
tsx
import React from "react";
import ReactDOM from "react-dom/client";
import { RouterProvider } from "react-router-dom";
import router from "./router";

ReactDOM.createRoot(document.getElementById("root") as HTMLElement).render(
  <React.StrictMode>
    <RouterProvider router={router} />
  </React.StrictMode>
);

3. 嵌套路由 ​

  • 路由配置项中children属性代表子路由
  • 需要注意的是，子路由默认是不渲染的，需要在父路由加载<Outlet />组件才会渲染子路由
tsx
import { createBrowserRouter } from "react-router-dom";

import Home from "../pages/home";
import Message from "../pages/message";
import About from "../pages/about";
import ErrorPage from "../pages/error-page";

const router = createBrowserRouter([
  {
    path: "/",
    element: <Home />,
    errorElement: <ErrorPage />,
    children: [ // 子路由
      {
        path: "message",
        element: <Message />,
      },
    ],
  },
  {
    path: "/about",
    element: <About />,
  },
]);

export default router;
tsx
import { Outlet } from "react-router-dom";

export default function Home() {
  return (
    <>
      <h1>Home</h1>
      // 加载渲染子路由
      <Outlet />
    </>
  );
}

7. 什么是虚拟 DOM？ ​

1. 概念 ​

虚拟 DOM（Virtual DOM）本质上就是一个 JavaScript 对象，通过对象的方式来表示 DOM 结构。

javascript
{
    type: "div" // 标签
    props: {
        className: 'title'
    }, // 属性
    children: [] // 子元素
    // ...
}

2. 优点 ​

  1. 性能更好：保证性能下限，在不进行手动优化的情况下，提供过得去的性能
  • diff 算法和批量处理策略, 将所有的 DOM 操作搜集起来，一起去改变真实的 DOM。比起自己一个个操作性能更好。
  1. 更强的兼容性：浏览器的兼容和跨平台兼容
  • React 基于虚拟 DOM 实现了一套自己的事件机制，并且模拟了事件冒泡和捕获的过程，采取事件代理、批量更新等方法，从而磨平了各个浏览器的事件兼容性问题
  • 对于跨平台，React 和 React Native 都是根据虚拟 DOM 画出相应平台的 UI 层，只不过不同的平台画法不同而已

8. React 中虚拟 DOM Diff 算法 ​

1. React渲染流程简述 ​

  • 初始化渲染：初始化渲染生成一个虚拟DOM树（旧树），根据旧树来渲染成真实DOM元素插入到页面上
  • 更新渲染：更新渲染也生成一个虚拟DOM树（新树），新树旧树进行比较（DOM Diff 算法），找出差异部分然后遍历更新差异部分的DOM元素，完成更新渲染。此时新树会成为下一次更新渲染的旧树。

2. Diff 算法优化 ​

由于Diff操作本身也会带来性能损耗，React文档中提到，即使在最前沿的算法中，将前后两棵树完全比对的算法的复杂程度为 O(n 3 )，其中n是树中元素的数量。 如果在React中使用了该算法，那么展示1000个元素所需要执行的计算量将在十亿的量级范围。这个开销实在是太过高昂。 为了降低算法复杂度，React的diff做了三个层面的优化：

  1. 只对同级元素进行Diff。如果一个DOM节点在前后两次更新中跨越了层级，那么React不会尝试复用他。
  2. 两个不同类型的元素会产生出不同的树。如果元素由div变为p，React会销毁div及其子孙节点，并新建p及其子孙节点。

3、 开发者可以通过** key 属性**来暗示哪些子元素在不同的渲染下能保持稳定。

3. Diff 算法详情 ​

当对比两棵树时，React 首先会从两棵树的根节点开始进行比较，然后在一一比较下面的子节点。

  1. 当对比节点是不同类型组件/元素，React 会拆卸原有的树并且建立起新的树，子节点就不会在比较了。
  2. 当对比节点是同一类型元素，React 会保留 DOM 节点，仅比对及更新有改变的属性，处理完当前节点之后，React 继续对子节点进行递归
  3. 当对比节点是同一类型组件，组件实例会保持不变，主要更新组件的 state 和 props，并且触发部分生命周期函数：UNSAFE_componentWillReceiveProps()、UNSAFE_componentWillUpdate() 、 componentDidUpdate() 和 render() 方法。调用 render() 时，diff 算法将在之前的 render 结果以及新的 render 结果中进行递归。
  4. 对子节点进行递归，使用 key 属性优化。

4. key 属性优化 ​

当递归 DOM 节点的子元素时，React 会同时遍历两个子元素的列表；当产生差异时，会打上标记将来更新。 在子元素列表末尾新增/删除元素时，更新开销比较小。例如： 更新前：

html
<ul>
    <li>111</li>
    <li>222</li>
</ul>

更新后：

html
<ul>
    <li>111</li>
    <li>222</li>
    <li>333</li>
</ul>

React 会先匹配两个 <li>1111</li> 对应的树，不变。然后匹配第二个元素 <li>222</li> 对应的树，不变。最后发现第三个元素 <li>333</li> 旧树没有，会执行插入操作。 但是如果需要将新增元素插入到开头，那么更新开销会比较大。例如： 更新前：

html
<ul>
    <li>111</li>
    <li>222</li>
</ul>

更新后：

html
<ul>
    <li>333</li>
    <li>111</li>
    <li>222</li>
</ul>

React 并不会意识到应该保留 <li>111</li> 和 <li>222</li> ，而是会删除旧的所有子元素，重建新的所有子元素。这种情况会带来性能问题。 为了解决上述问题，React 引入了 key 属性。当子元素拥有 key 时，React 使用 key 来匹配原有树上的子元素以及最新树上的子元素。例如： 更新前：

html
<ul>
    <li key="1">111</li>
    <li key="2">222</li>
</ul>

更新后：

html
<ul>
    <li key="3">333</li>
    <li key="1">111</li>
    <li key="2">222</li>
</ul>

现在 React 知道只有带着 3' key 的元素是新元素，带着 '1 以及 '2' key 的元素仅仅移动了。 实际开发中 key 的值一般用数据的 id。这个 key 不需要全局唯一，但在列表中需要保持唯一且更新前后稳定。

问题：key 使用 index 存在什么问题？ 更新前：

html
<ul>
    <li key="0">111</li>
    <li key="1">222</li>
</ul>

更新后：

html
<ul>
    <li key="0">333</li>
    <li key="1">111</li>
    <li key="2">222</li>
</ul>

React 会先匹配两个 key="0" 对应的树，发现值不一样，标记更新。再匹配两个 key="1" 对应的树，发现值不一样，标记更新。最后 key="2" 旧树没有，需要插入。这样所有元素都没能复用，全部需要更新，更新性能较差。
